Tag System

type: component components: redis deployment_types: helm resource_types: storage
Design: One pattern per file → Enables precise tag-based retrieval → No embedding complexity needed

Conversion Philosophy

80/20 Approach

  • 80% Pattern Matching - Automatic application of proven patterns
  • 20% Custom Decisions - User input for project-specific choices

Minimal Changes

  • Conditional Mode: Add RHOAI support without breaking NVIDIA
  • Helm: openshiftMode: false
  • oc apply: OPENSHIFT_MODE env var

Right Tool for the Job

📓 Notebooks → oc apply
Simple, self-contained deployments
⎈ Multi-service → Helm
All resources templated, values.yaml configuration

Key Design Decisions

✓ Tag-Based Scoring (No Embeddings)

Simple, fast, transparent - sufficient for our scale

✓ On-Demand Loading

Load only top-scored relevant knowledge → Prevent context overflow

✓ Multiple Approaches Per Component

Approach A/B with "when to use" guidance

✓ Guardrails Checklist

Coverage: GPU, storage, security, networking, init order, DNS, secrets

✓ Red Hat Docs Integration

Context7 MCP tool for knowledge gaps

⚠️ Challenge: Context Management

Complex blueprints need adaptive loading (discovered in Test 2)
